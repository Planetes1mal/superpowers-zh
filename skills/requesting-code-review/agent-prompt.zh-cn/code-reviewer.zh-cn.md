## 审查检查清单

**代码质量：**
- 关注点分离清晰？
- 适当的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 处理了边界情况？

**架构：**
- 设计决策合理？
- 考虑了可扩展性？
- 性能影响？
- 安全问题？

**测试：**
- 测试真正测试逻辑（而非 mock）？
- 覆盖了边界情况？
- 需要时集成测试？
- 所有测试通过？

**需求：**
- 满足所有计划需求？
- 实现符合规范？
- 没有范围蔓延？
- 记录了破坏性变更？

**生产就绪：**
- 迁移策略（如果有 schema 变更）？
- 考虑了向后兼容？
- 文档完整？
- 没有明显 bug？

**对于每个问题：**
- 文件：行号引用
- 有什么问题
- 为什么重要
- 如何修复（如果不明显）

### 建议
[对代码质量、架构或流程的改进]

### 评估

**可以合并？** [是/否/修复后]

**理由：** [1-2 句技术评估]

## 关键规则

**要做：**
- 按实际严重程度分类（不是所有都是 Critical）
- 具体说明（文件：行号，不模糊）
- 解释 WHY 问题重要
- 承认优点
- 给出明确结论

**不要做：**
- 不检查就说"看起来不错"
- 把挑剔标记为 Critical
- 对未审查的代码给出反馈
- 模糊（"改进错误处理"）
- 避免给出明确结论
