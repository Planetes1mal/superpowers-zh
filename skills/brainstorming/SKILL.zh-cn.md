# 将想法头脑风暴为设计

## 概述

通过自然的协作对话，帮助将想法转化为完整的设计和规格说明。

首先了解当前的项目上下文，然后逐一提问以完善想法。一旦理解了要构建的内容，就以小块（200-300字）呈现设计，并在每个部分后确认是否看起来正确。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近的提交）
- 逐一提问以完善想法
- 尽可能使用多选题，但开放式问题也可以
- 每条消息只提一个问题——如果一个话题需要更多探索，将其拆分为多个问题
- 专注于理解：目的、约束条件、成功标准

**探索方法：**
- 提出2-3种不同的方法及其权衡
- 以对话方式呈现选项，附上你的建议和理由
- 首先提出你推荐的选择并解释原因

**呈现设计：**
- 一旦你认为自己理解了要构建的内容，就呈现设计
- 将其分解为200-300字的章节
- 在每个章节后询问目前看起来是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果某些内容不合理，准备好返回并澄清

## 设计完成后

**文档：**
- 将验证过的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely skill
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问："准备好开始实现了吗？"
- 使用 superpowers:using-git-worktrees 创建隔离的工作空间
- 使用 superpowers:writing-plans 创建详细的实现计划

## 关键原则

- **一次一个问题** - 不要用多个问题压倒用户
- **优先使用多选题** - 在可能的情况下，比开放式问题更容易回答
- **严格遵循 YAGNI** - 从所有设计中移除不必要的功能
- **探索替代方案** - 在确定之前始终提出2-3种方法
- **增量验证** - 分章节呈现设计，逐一验证
- **保持灵活** - 当某些内容不合理时，返回并澄清
